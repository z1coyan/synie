**产品说明:** [docs/产品文档/系统管理.md](../../docs/产品文档/系统管理.md)（用户/角色/权限矩阵节）

**术语澄清：**「岗位」不是本系统的领域概念。系统里只有角色（用户可配置的权限码集合）与两个旗标（`super_admin`、`all_companies`）；「会计」「销售员」等岗位只是生产环境里配置出的角色数据。因此本 spec 验证的不是某个虚构岗位包，而是系统不变量本身——**任意权限码集合 × 公司授权集合的主体，可见集恰好等于应得集，一条不多一条不少**。

---

## Problem Statement

系统的权限执法完全在服务端（Ash policy 的功能权限 + 公司数据权限，前端按钮显隐不是安全边界），执法面已经很大：54 个权限组、约 240 个权限码、43 个按公司隔离的资源、3 个 HTTP 数据出口（GraphQL、文件下载、打印导出）。

既有测试是**机制级**的：权限匹配语义、fail-closed、公司过滤各有单测，个别资源有授权感知的功能测试。但没有任何测试回答老板真正关心的问题——**「不同岗位的人是否会获得不属于他的数据」**——即全量资源 × 全量主体形态的系统性验证。一个资源忘挂公司过滤、一个出口漏判宿主权限（#41 审计修复的事故形态），现有测试网都可能漏过。且每新增一个资源，这张网不会自动变大。

## Solution

三层防线，全部自动化、随 CI 常跑（浏览器层除外）：

1. **API 层权限矩阵扫描（主体）**：以真实 HTTP 管线（含 token 验证）为唯一测试缝，用权限目录反射生成的合成极值主体（无码、最小授权+公司甲、只授公司乙、`all_companies`、`super_admin` 等），对一个预先建好的「双公司夹具世界」逐资源发起读写请求，双向断言"可见集恰好等于应得集"。新资源进权限目录即自动进矩阵。
2. **静态守卫（抓"忘了挂"）**：内省断言每个进目录的资源都挂了功能权限校验，每个带公司字段的资源都挂了公司读过滤与写校验；夹具世界的覆盖表与权限目录 diff，缺一个构造函数 CI 即红。
3. **浏览器薄冒烟（辅助）**：脚本化 Playwright 单场景——超管走真实管理动线临时建「演示会计」角色并授权，以该角色登录，断言菜单/页面/直连 URL 的可见性符合授权。验证前端消费权限的端到端动线，不承担覆盖率。

另有两组定向场景集：**R3 出口**（文件下载、打印导出的跨公司与裸文件越权）与 **R2 定点**（表格元数据反射的 fail-closed 降级）。

## User Stories

1. As a 只授权公司甲的业务用户, I want 列表查询永远看不到公司乙的记录, so that 跨公司数据不泄露
2. As a 只授权公司甲的业务用户, I want 拿到公司乙记录的 id 也无法按 id 查到它, so that 泄露的 id 不能变成泄露的数据
3. As a 只授权公司甲的业务用户, I want 聚合列/计算列的数值也只统计公司甲, so that 汇总数字不间接泄露他司经营数据
4. As a 持有某资源 read 码的用户, I want 确实能读到本公司该资源的记录, so that 权限系统不把人锁死（正向断言）
5. As a 不持有某资源 read 码的用户, I want 对该资源的一切查询被拒, so that 无功能授权即无数据
6. As a 未登录者, I want 一切数据出口返回未认证, so that 匿名不可达任何数据
7. As a `all_companies` 用户, I want 在持码资源上看到所有公司的记录, so that 跨公司旗标语义正确
8. As a `super_admin`, I want 全部资源全部公司可读可写, so that 超管兜底语义正确
9. As a 持码但未获任何公司授权的用户, I want 在公司隔离资源上得到空集, so that 公司授权 fail-closed
10. As a 只授权公司甲的用户, I want 在公司乙下创建记录被拒, so that 写侧不能跨公司落数据
11. As a 只授权公司甲的用户, I want 修改/删除公司乙的记录被拒, so that 写侧不能篡改他司数据
12. As a 不持有写码的用户, I want create/update/destroy mutation 全部被拒, so that 无码不可写
13. As a 上传了裸文件（未挂宿主）的用户, I want 只有我自己和超管能下载它, so that 裸文件不成为越权通道
14. As a 只授权公司甲的用户, I want 下载公司乙单据的附件被拒, so that 文件出口遵守公司隔离
15. As a 不持有宿主资源 read 码的用户, I want 下载该宿主的附件被拒, so that 文件出口遵守功能权限
16. As a 普通用户, I want 把他人上传的裸文件补挂到我可见的宿主被拒, so that 补挂不能洗白越权下载
17. As a 只授权公司甲的用户, I want 打印导出公司乙的单据被拒, so that 批量二进制出口不绕过数据权限
18. As a 不持有 print/export 码的用户, I want 打印/导出请求被拒, so that 出口遵守功能权限
19. As a 任何用户, I want 对不在可打印清单内的资源请求打印被拒, so that 出口面不被反射扩大
20. As a 无目标资源 read 码的用户, I want 表格元数据里的外键/多态引用降级为普通列, so that 元数据不泄露关联资源的入口
21. As a 任何用户, I want 请求白名单外资源的表格元数据得到报错, so that 反射机制不成为探测面
22. As a 开发者, I want 新资源进权限目录后自动进入矩阵扫描, so that 覆盖零维护、不靠自觉
23. As a 开发者, I want 新资源没写夹具构造函数时 CI 变红, so that 夹具世界不悄悄落后于目录
24. As a 开发者, I want 新资源忘挂权限策略/公司校验时被静态守卫拦下, so that 最常见的事故形态在合并前暴露
25. As a 开发者, I want 声明了 read 的资源缺席表格白名单且无豁免时 CI 变红, so that 读侧枚举源不漏资源
26. As a 权限管理员, I want 建角色→授权→指派用户/公司→登录生效这条动线被冒烟覆盖, so that 权限管理本身端到端可信
27. As a 演示会计（冒烟角色）, I want 登录后菜单只出现授权模块, so that 前端可见性与授权一致
28. As a 演示会计（冒烟角色）, I want 直接输入未授权页面的 URL 被拒或得到空态, so that 绕过菜单也无法到达数据
29. As a 维护 CI 的人, I want 矩阵与守卫随现有后端 job 常跑, so that 每次 push/PR 都有越权回归保证
30. As a 维护 CI 的人, I want 浏览器冒烟不进 PR CI 而是一键本地/按需跑, so that PR 时长与脆性不被拖累

## Implementation Decisions

以下决策来自一轮完整的 grilling，编号沿用会话中的类别词（R1 读泄露 / R2 关联泄露 / R3 出口泄露 / W1 写越权）。

### 射程

- R1 + R3 + W1 全量进射程；R2 只做定点断言（元数据泄露风险低，且外键速览的数据面已被 R1 的按 id 查询覆盖）。
- 领域动作（审核/发货/对账等）**不进全量矩阵**：合法调用姿势各异，反射调用不可行；其权限门与通用动作同机制，机制有单测，各动作已有授权感知的功能测试。矩阵的增量价值在数据隔离维度，写侧通用三件套（跨公司建/改他司/删他司）足以验证公司写校验的挂载。

### 主体轴（actor）

- 合成极值主体由权限目录反射生成，不手写清单。每资源至少覆盖：无码者、恰好持 `该资源:read` + 公司甲授权的最小主体、同码但只授公司乙的补集主体、持码但零公司授权者、`all_companies` 持码者、`super_admin`。写侧同理以最小写码构造。
- 现实岗位包不进 API 矩阵（断言虚构配置而非系统不变量，且维护成本随资源数线性增长）；仅浏览器冒烟用一个现实角色，取其可读性。

### 数据轴（夹具世界）

- 集中式「双公司夹具世界」：公司甲、公司乙，每个公司隔离资源在两家公司各一条合法记录，全局资源各一条。构造函数逐资源手写（`authorize?: false`，尽量搬现有 domain fixtures），这是本方案最大的一次性人工投入。
- 每资源挂**应得集声明**（expected-visibility）：默认规则"公司匹配即应得"；特例（裸文件仅上传者、审计日志无公司行放行、全局资源有码即读）以显式声明覆盖，矩阵断言以声明为 oracle，特例不硬编码进断言循环。
- **完整性守卫**：权限目录资源清单 diff 夹具世界覆盖表，缺席即红（模式照抄现有目录清单硬编码守卫）。

### 操作轴

- 读侧枚举源 = 表格元数据白名单（它本身就是前端数据入口的权威清单）：每资源打 list 查询与按 id 查询，聚合列随 list 顺带断言。配守卫：目录中声明 read 的资源缺席白名单且无豁免声明 → 红。
- 写侧 = 通用三件套：跨公司 create、update 他司记录、destroy 他司记录，外加无码 mutation 全拒。

### 断言语义

- **双向"恰好等于"**：最小主体必须读到本公司那条（正向），且恰好读不到乙司那条（负向）。只测负向会让"全部拒绝"的坏系统通过——正向断言守住"权限系统没把人锁死"这半边。

### 载具与缝

- 唯一主缝 = 真实 HTTP 管线（Phoenix ConnCase 打 GraphQL 与 REST endpoint，带真实 Bearer token），token 验证、上下文 plug、401 路径全在射程内；不用进程内直调，也不起真实 TCP 服务器。矩阵套件落在 web 应用的测试内，进现有 CI 后端 job。
- 辅缝 = 脚本化 Playwright 浏览器冒烟：setup 阶段以超管 token 走真实 GraphQL 管理动线（建角色→同步授权→指派用户与公司），断言完丢弃；演示角色**不进迁移种子**（种子会发货到生产、稀释"内置=迁移种子"约定，且临时造顺路覆盖管理动线）。

### 静态挂载守卫（矩阵之外的第三层）

- 内省断言：凡声明权限前缀的资源必挂功能权限策略；凡带公司字段的资源必挂公司读过滤与公司写校验。动态矩阵抓"行为漏"，此守卫抓"忘了挂"。

### 已知实现皱褶（实现时定，不在此锁死）

- 夹具世界建一次 vs 每模块建一次，受 SQL Sandbox 与 `setup_all` 相性约束；必要时退化为每模块建一次。
- umbrella 下核心应用的 test/support 对 web 应用测试不可见，世界构造器的物理落点需选两边都够得着的位置。

## Testing Decisions

本 spec 的交付物本身就是测试，此节说明"测试的测试观"：

- **只断言外部行为**：HTTP 状态、GraphQL data/errors、返回的数据集合。不断言内部实现（不数查询条数、不窥探 policy 内部）。oracle 来自每资源的应得集声明，**不得**在测试里重新实现一遍过滤逻辑来对答案。
- **被测面**：GraphQL 全部白名单资源的读写、文件下载/补挂 endpoint、打印导出 endpoint、表格元数据反射、前端消费动线（冒烟）。
- **既有先例（prior art）**：GraphQL 授权测试与公司过滤单测（含真实 `build_actor` 链路与直构 struct 两种主体构法）、权限目录清单硬编码守卫、文件/打印控制器测试、授权夹具模块（`user!/role!/grant!/assign!/company!/grant_company!` 一族）。矩阵是它们的外积推广，不是新范式。
- **失败可读性**：矩阵断言失败信息必须点名"哪个资源 × 哪个主体形态 × 哪个方向（正/负）"，否则 54 组资源的循环断言炸起来无法定位。

## Out of Scope

- 领域动作（审核/发货/对账/红冲等）的全量矩阵扫描——留在各自功能测试。
- 现实岗位包进 API 矩阵；任何"标准岗位模板"产品功能。
- R2 全量矩阵（只做定点）。
- 浏览器冒烟进 PR CI；nightly 化留作后议（标注为可复议决策）。
- 字段级/行级更细粒度权限的测试（系统本身没有该维度）。
- 性能与压测、生产配置审计工具、演示角色种子数据。

## Further Notes

- 历史事故形态参考：#41 的打印/文件出口审计修复（裸文件下载限上传者、附件两段权限门）正是 R3 类漏洞——本 spec 的 R3 场景集就是给这类修复上回归保险。
- 浏览器冒烟依赖 HeroUI Pro token 与前后端同起，脆性天然高于 API 矩阵；如果它先烂掉，砍掉它不影响主体防线——这是把覆盖率全部压在 API 层的另一个理由。
- 矩阵规模粗估：54 组资源 × ~6 主体形态 × 读写若干操作 ≈ 千余次请求；ConnCase + Sandbox 量级可承受，若超时预算再谈分片。
