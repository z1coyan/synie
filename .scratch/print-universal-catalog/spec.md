# Spec: 打印模板全资源自动接入（字段清单内省派生）

**Status:** ready-for-agent
**Feature slug:** `print-universal-catalog`
**Depends on:** `print-template-master`、`print-xlsx-engine`、`print-document-pipeline`（v1 已交付）
**ADR:** [docs/adr/2026-07-23-print-template.md](../../docs/adr/2026-07-23-print-template.md)（初版）、[docs/adr/2026-07-23-print-universal-catalog.md](../../docs/adr/2026-07-23-print-universal-catalog.md)（本规格定案）
**Domain terms:** 打印模板、模板占位符、明细模板行、对手（Party）、标准动作、权限目录（见 `CONTEXT.md`）
**产品说明:** [docs/产品文档/系统管理.md](../../docs/产品文档/系统管理.md)「打印模板」

---

## Problem Statement

业务上希望任意资源（各业务单据、主数据）都能挂 Excel 打印模板。v1 只支持销售订单与销售发货单两种：每接入一个资源，开发要手写两处——字段清单逐字段抄一遍、装配器逐资源写取值子句。系统约 103 个资源，逐个手写不现实，新业务单据上线即被打印能力卡住。模板作者还面对两个别扭：关联字段是拍平键（`company_name`）而非直觉的路径（`company.name`）；循环能力只有引擎写死的一种明细行，多子表资源（如 BOM 的配料行/副产品行/工艺路线行）无法表达。

## Solution

字段清单改为由 Ash 资源内省**自动派生**，全部资源零代码接入打印模板体系；资源级定制全部不要，定制只发生在 xlsx 模板文件上。引擎内置三条对全部资源统一适用的语义规则：belongs_to 关联按路径展开一层、枚举值统一映射中文标签、`party_type`/`party_id` 约定统一解析出对手名称。占位符一律路径式（`${company.name}`、`${items.material_name}`），与明细循环统一为「关系.字段」语法；循环区泛化为每个 has_many 按关系名寻址，同一模板允许多个顺序循环区，嵌套循环上传即拒。主键/外键/时间戳等技术列不进清单。旧拍平键模板须重新上传，上传校验逐个点名拒掉。上传即校验、版式主权在 Excel、打印/导出同引擎两产物等 v1 条款全部照旧。

## User Stories

1. As a 业务用户, I want 对任意资源的记录发起模板打印, so that 不再只有销售订单/发货单两种能打
2. As a 业务用户, I want 对任意资源做模板导出（xlsx）, so that 打印与导出产物版式恒一致（沿 v1）
3. As a 业务用户, I want 无可用模板时提示先去系统管理上传, so that 我知道下一步去哪（沿 v1，不回退列表行打印）
4. As a 业务用户, I want 批量打印/批量导出照 v1 行为（100 条上限、块间分页/多 sheet）, so that 既有操作习惯不变
5. As a 模板制作者, I want 用 `${company.name}` 这类路径占位符取关联字段, so that 占位符含义自描述、不用背拍平键对照表
6. As a 模板制作者, I want 任意 belongs_to 关联可按路径展开一层（如 `${warehouse.name}`、`${employee.code}`）, so that 取关联口径不必等开发加字段
7. As a 模板制作者, I want 关联只能下一层（`${company.name}` 合法、再下钻非法）, so that 语法边界明确、校验能点名
8. As a 模板制作者, I want 枚举字段（如状态）打印出中文标签（如「已审核」）, so that 中文 ERP 的纸上不出现英文枚举原文
9. As a 模板制作者, I want 带 `party_type`/`party_id` 的单据自动有对手名称占位符可用, so that 订单上能印出客户名称而不是 UUID
10. As a 模板制作者, I want 明细循环用 `${items.字段}`（关系名寻址）, so that 与 v1 习惯一致
11. As a 模板制作者, I want 多子表资源的每个 has_many 各引一个循环区（如 BOM 的配料行、副产品行、工艺路线行各占一段）, so that 一份模板能印全多组子行
12. As a 模板制作者, I want 每个循环区都有 `_seq` 行序号保留字, so that 各组子行都能编号
13. As a 模板制作者, I want 嵌套循环（循环区目标资源自己的 has_many，如条目内套价格档）上传时被拒绝并点名, so that 不支持的版式在上传时就知道，而不是打印出废纸
14. As a 模板制作者, I want 某循环区数据为 0 行时该循环区模板行整行删除, so that 空子表不留残行（沿 v1 明细行语义，逐循环区适用）
15. As a 模板制作者, I want 主键、外键、时间戳不出现在字段清单里, so that 清单干净、不会误把 UUID 印上纸
16. As a 模板制作者, I want 资源上已定义的计算字段/聚合（如 `gross_total`）直接可用, so that 金额链口径与界面一致
17. As a 模板制作者, I want 无 has_many 的资源（如工资单、物料）退化为纯字段替换, so that 主数据卡片类模板也能做
18. As a 模板制作者, I want 字段清单面板直接显示路径名作为标签, so that 我看到的就是我要抄进 Excel 的，无中间翻译层
19. As a 模板制作者, I want 空值占位符渲染为空字符串, so that 空字段不印出乱七八糟的原文（沿 v1）
20. As a 系统管理员, I want 模板管理页的资源选择覆盖全部资源（不再硬编码两个）, so that 任何资源都能上传模板
21. As a 系统管理员, I want 资源在管理页以权限目录的中文资源名分组展示, so that 找资源不靠背资源码
22. As a 系统管理员, I want 上传即校验照旧——未知字段拒绝保存并逐个点名, so that 错模板活不到打印时（沿 v1）
23. As a 系统管理员, I want 含旧拍平键（如 `${company_name}`）的存量模板重新上传时被拒并点名该键, so that 迁移靠校验自然完成、无静默失效
24. As a 系统管理员, I want 只收 .xlsx、页面设置警告不拦截等 v1 校验行为不变, so that 上传规则无意外
25. As a 权限管理员, I want 打印/导出/批量打印入口仍随标准动作逐资源启用, so that 权限矩阵口径不变（接入是能力，入口开关不变）
26. As a 开发, I want 新增 Ash 资源时打印侧零代码即获得接入能力, so that 新单据上线不再被打印排期卡住
27. As a 开发, I want 枚举中文标签直接从枚举类型自身取值（本仓库枚举统一 `values: [atom: "中文"]` 写法）, so that 新增枚举值只动枚举定义一处，v1 装配器里的逐资源手写映射删除
28. As a 开发, I want 字段派生规则（技术列排除、路径展开、循环区）全部资源一刀切, so that 没有资源级特殊分支要维护

## Implementation Decisions

- **字段清单模块（FieldCatalog）改为派生，核心接口维持**：`resources/0`、`get/1`、`field_names/1`、`validate_placeholders/3` 维持，调用方（模板校验、管理页 API）随之小改（`get/1` 返回演进为 `{fields, loops}`；`validate_placeholders/3` 第三参由明细字段列表演进为按首段归组的 nested map）。内部从手写表改为按 Ash 内省派生。`resources/0` 返回权限目录覆盖的全部资源码（替代现状的手写两键），循环区信息经 `get/1` 暴露。（实施修订：`item_names/1` 删除——多循环区语义下"items 的字段集"不再是公共面，无调用方，恢复它是死代码；枚举标签不再建集中映射表——本仓库枚举类型自带 `values: [atom: "中文"]` description，直接经 `type.description/1` 取值，零维护且随类型同步，v1 手写映射随之删除。）
- **派生规则（全部资源一刀切）**：
  - 标量属性进清单，排除：主键 `id`、外键 `*_id`、`inserted_at`/`updated_at`。
  - 资源上已定义的计算字段与聚合进清单（领域口径的一部分，非打印专用派生）。
  - belongs_to 关联展开一层：`<关系名>.<目标资源标量属性>`，目标侧同样排除技术列；不支持第二层下钻。
  - 枚举属性值统一经枚举类型的 `description/1` 渲染中文标签（本仓库枚举统一 `values: [atom: "中文"]` 写法，标签随类型自带，无需独立映射表；v1 装配器的逐资源手写映射收编删除）。
  - 资源同时具有 `party_type`/`party_id` 时派生对手名称占位符：按 party_type 值映射目标资源（客户/供应商/内部公司/员工，同 Party 词条四类）取其名称；解析逻辑收敛为引擎统一规则，删除 v1 装配器里的逐资源 party 查询代码。
  - 每个 has_many 成为一个**循环区**，以关系名寻址；循环区字段按同一套规则派生（含其 belongs_to 一层路径）；循环区目标资源自身的 has_many 不得再作循环（嵌套），`validate_placeholders/3` 检出即拒并点名。`_seq` 为各循环区引擎保留字。
- **装配器（DocBuilder）改为通用装配**：按派生清单对任意资源统一 `Ash.load`（belongs_to 一层 + 全部 has_many 循环区 + party 目标）并取值，产出 `%{fields, items}` 结构泛化为「头字段 + 各循环区行集」；v1 两个逐资源子句删除。格式化纪律不变：取值一律转字符串、空值归空串、日期/数字显示格式仍由单元格 Excel 格式承载（引擎不发明格式化语法）。
- **渲染器（Renderer）占位符解析泛化**：按「关系.字段」分段——无点为头字段直取；有点且前缀为某 has_many 关系名即循环区占位，该行成为该循环区的模板行。行复制逻辑按循环区分组执行：多循环区各占自己的行段顺序展开，`_seq`、下方行与 mergeCell 顺移语义同 v1；某循环区 0 行删其模板行。未知键渲染为空字符串照旧（容错从宽，校验在上传端把关）。
- **旧拍平键不兼容、不做迁移代码**：派生清单不再含 `company_name` 等拍平键，存量模板重新上传时被既有校验以「未知头字段」点名拒掉——迁移通道即校验本身。已存模板渲染出空白属预期（打印是瞬时动作、不留痕，重传即恢复）。
- **模板校验（Template）**：占位符抽取与拒绝文案机制不变，校验源换为派生清单；新增嵌套循环检出（点名占位符与其穿到的循环区）。
- **前端模板管理页**：资源下拉从硬编码两项改为动态（权限目录资源清单，中文名分域展示）；字段清单面板按「头字段 / 各循环区」分组，条目直接显示路径名（无字段级中文标签）；field-catalog API 返回结构从 `{fields, items}` 演进为 `{fields, loops: [{name, fields}]}`，管理页随之调整。
- **入口与权限**：`print`/`export`/`batch_print` 标准动作启用机制与权限码不变；本规格只保证任何资源被启用后打印链路立即可用，不批量打开任何资源的入口。
- **无数据库 schema 变更**：派生为运行时内省；枚举标签表为代码常量。
- **文档同步（交付时）**：`CONTEXT.md` 改写「打印模板」「模板占位符」「明细模板行」三词条（明细模板行泛化为循环区概念）；`docs/产品文档/系统管理.md`「打印模板」节同步；`docs/adr/` 补新 ADR 记录本次修订（手写清单 → 内省派生、拍平键 → 路径式、单循环 → 多循环区）。

## Testing Decisions

- **好测试标准**：只测外部行为——上传接口的接受/拒绝与点名文案、导出文件的行内容、派生清单模块公开接口的返回；不断言派生/装配的内部中间结构。
- **接缝一（集成，既有）**：`TemplateAndExportTest`（真库沙箱 + 真上传 + 走完整模板校验与导出管线）。覆盖：
  - 销售订单用路径键（`${company.name}` 等）上传并导出，断言行值（含枚举中文标签、对手名称、计算字段）；
  - 任选一个 v1 未接入资源（建议一个无 has_many 主数据 + 一个多 has_many 资源如 BOM）验证零代码接入：上传 → 校验通过 → 导出断言；
  - 旧拍平键模板上传被拒且报错点名 `company_name`；
  - 嵌套循环模板上传被拒并点名；
  - 多循环区模板导出断言各循环区行数与 `_seq`。
- **接缝二（引擎单元，既有）**：`RendererTest`（PrintingFixture 内存构造 xlsx，不落库）。覆盖：路径占位符替换（含 sharedStrings 改写 inline）、混排与未知键填空、多循环区行复制 + `_seq` + 下方行/mergeCell 顺移、0 行循环区整行删除。
- **派生规则**：在 FieldCatalog 既有公开接口上单元测——技术列排除、计算字段进、belongs_to 一层路径形状、party 名称派生、循环区列表、`validate_placeholders/3` 拒未知/嵌套。不新增测试基建。
- **DocBuilder 不单开接缝**：通用装配行为经接缝一覆盖。
- **Prior art**：`renderer_test.exs`、`template_and_export_test.exs`、`pdf_converter_test.exs` 及 `PrintingFixture`（xlsx 构造/读回工具）全部沿用；LibreOffice 转换行为不在本规格重测。

## Out of Scope

- 嵌套循环（循环区内再套循环区，如报价条目内嵌价格档）——上传校验明确拒绝。
- 多层关联下钻（`${company.base_currency.code}` 这类第二层）。
- 字段级中文标签的维护与展示（标签=路径名本身）。
- 旧拍平键的兼容层、别名或迁移工具（重传即迁移）。
- 资源级字段定制：隐藏字段、改字段名、打印专用派生字段。
- 图片类占位符（图纸/Logo，沿 v1 排除）。
- 批量打开各资源打印入口（入口启用仍逐资源走既有机制）。
- PDF 转换、批量打印合并等 v1 引擎行为的变更。

## Further Notes

- 本规格来自 2026-07-23 拷问访谈定案，与同日 v1 ADR 的关系是**修订而非推翻**：上传即校验、版式主权在 Excel、一条引擎两个产物、降级切割、模板全局主数据不留痕等条款全部继承。
- 「循环区」是「明细模板行」的泛化：v1 的 `${items.*}` 是循环区的一个特例（关系名恰为 `items`），交付时词条相应改写。
- 枚举标签表虽小但属新公共物：后续其它需要枚举中文标签的展示场景可复用，本规格不强行推广。
