# 员工参保类型：原子险种多选枚举，数组列筛选走伴生 calculation

需求原文是四个「套餐」值（社保工伤险/社保五险+公积金/社保三险/平安企业工伤险），拷问后拆成八个原子险种（社保工伤/失业/医疗/养老/生育、公积金、商保工伤/商保医疗）固定枚举，员工档案单字段 `{:array, enum}` 多选任意组合、不设互斥——「五险一金」「三险」只是勾选组合，换档过渡期两档并存不被系统挡；商保值不带保司/保单信息（现仅投平安一家，保单台账是另一需求）；当前状态快照不做参保区间（变更靠审计日志）；纯档案字段不参与工资计算，将来社保扣款联动的预留就是原子化本身——扣款引擎按险种读取，无需改档案结构。可维护字典表方案被否（险种低频变动，加值改一行枚举；新资源+关联表+权限+页面过重），按保司拆值被否（枚举随保司膨胀）。技术上这版 AshGraphql 硬排除数组属性的 filter 字段（`filterable?({:array,_}) → false`，`has` 算子进不了自动生成的 filter 输入），「包含/不包含」筛选立了伴生 calculation 约定：数组属性配同名 `<attr>_has` 布尔 expr calculation（参数名固定 `type`），前端拼 `{<attr>Has: {input: {type: X}, eq: true/false}}`，「包含任一」or 组合、「不包含」and 组合；GridMeta 对枚举数组列 fail-closed——伴生 calc 缺失即标不可筛，防 filterable 列拼出 schema 里不存在的 filter 字段整查报错。每险种一布尔列的替代方案被否：表单/表格八列噪音大、筛选入口碎、加险种要迁移。
